# 0001: Cloudflare Runtime And Data Platform

Status: Accepted

Date: 2026-06-05

## Context

The current app is a static Next.js export that reads generated JSON from
`public/data`. Historical ingest commits generated payloads back to the repo.
Live data is generated by GitHub Actions and force-pushed to a `live-data`
branch, then fetched by the browser from `raw.githubusercontent.com`. The
current Cloudflare Worker only dispatches a GitHub Actions workflow on a cron.

This works for the prototype, but it makes git and GitHub-hosted raw files part
of the runtime data platform. That is not the desired long-term boundary.

## Decision

Use GitHub for source control and CI. Move production hosting and runtime data
toward Cloudflare:

- Cloudflare Pages serves the static frontend.
- Cloudflare Worker API becomes the browser-facing data boundary.
- R2 stores raw, normalized, generated, and derived payload objects.
- D1 stores catalog metadata, source runs, dataset availability, analysis
  indexes, and data quality summaries.

The migration should be incremental. The current static app and generated JSON
contracts should remain working until the Worker API and Cloudflare-backed
storage are ready to replace them.

## Consequences

- New browser data access should be designed around Worker API endpoints, not
  direct reads from GitHub raw URLs or R2 object URLs.
- R2 object names and D1 table layouts are internal implementation details
  unless explicitly promoted into an API contract.
- Source adapters, normalized datasets, derived analyses, and visualisations
  must remain separate concepts.
- Existing `public/data` files can be treated as compatibility outputs during
  migration.
- The current Worker cron dispatcher should not be broken casually; it is an
  operational component until replaced.

## Target API Surface

- `GET /api/catalog`
- `GET /api/days`
- `GET /api/latest`
- `GET /api/day/:date`
- `GET /api/live`
- `GET /api/analyses`
- `GET /api/analyses/:id`

## Non-Goals

- No immediate whole-app rewrite.
- No immediate removal of `public/data`.
- No requirement to stabilise unrelated prototype behavior as part of this
  decision.
- No direct frontend coupling to source-specific raw data formats.
