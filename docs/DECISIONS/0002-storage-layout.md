# 0002: R2 Object Layout And D1 Catalog Schema

Status: Draft

Date: 2026-06-05

## Context

The prototype writes generated JSON files to `public/data/` and commits them to
git. The Cloudflare migration needs a storage boundary that keeps payload bodies
out of git while preserving the existing static compatibility files until the
Worker API is ready.

## Decision

Use R2 for payload bodies and D1 for queryable metadata. R2 keys and D1 tables
are internal implementation details owned by the storage module; the browser
only sees Worker API responses.

R2 key layout:

```text
raw/<source>/<run-id>/<original-filename>
dataset/<dataset-id>/<date>.json
compat/day/<date>.json
compat/index.json
compat/latest.json
compat/live.json
analysis/<analysis-id>/<version>.json
```

Rules:

- Keys are lowercase path segments except stable ids that already contain
  documented separators.
- `run-id` is generated by the writer and recorded in D1.
- `dataset-id` identifies the normalized dataset family, not the source file.
- `compat/*` mirrors current frontend JSON during migration only.
- R2 object keys are not public API fields unless a later decision explicitly
  promotes them.

D1 owns:

- source definitions and source runs
- normalized dataset definitions and date availability
- analysis definitions and availability
- data quality summaries
- schema/contract versions

## Consequences

- Ingest and analysis writers can store raw, normalized, compatibility, and
  derived payloads without coupling frontend views to storage paths.
- Worker API endpoints can serve `/api/days`, `/api/latest`, `/api/day/:date`,
  `/api/live`, `/api/analyses`, and `/api/catalog` from storage while preserving
  current compatibility shapes.
- CI/deployment must provision the R2 bucket and D1 database and bind them to
  the Worker before this storage module can be active in production.
